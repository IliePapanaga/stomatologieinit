3.2 Permanent Job Management

You can create a service template that can be re-used when deploying a VM with applications. You can also clone and fix a template (PLACEHOLDER) and/or add a service provider's template (PLACEHOLDER). You can add an application template, too. The logical difference between an application and service template is that services one are intended for Java, Tomcat, etc., while the application templates are geared towards apps like MS Word, etc. to be deployed on VMs. Note that these two types of templates are functionally and technically identical.

Note: You must specify at least the properties of a service template, its deployment profile, and operations before you can save it in P3C.
Remember: Before adding a composite template, you must add a regular, plain template first (a composite template can include a range of plain templates). Note that a plain template becomes available within a composite template only after you have provisioned and published it.
  1. Click Design Studio.
  2. Click the Service Templates tab, then Add Template, Properties, and enter the following:
    • Service Template Name - an arbitrary name.
    • Version - click Browse, and specify the following:
      • Service Type - the service type you had set up.
      • Major, Minor, Maintenance ID, Build # - alphanumeric fields you must use to configure the template's major and minor versions, etc., in order to differentiate it from the others.
      • Key - the template's short name: P3C automatically verifies its uniqueness.
      • Display Name - PLACEHOLDER.
      • Icon - select an image file to be displayed next to the template for identification purposes.
  3. Configure media folders, media files, topology, and service parameters.
  4. Click Operations, then Add Operation and Properties.
    Attention: Once you have provisioned and published the template, you will not be able to edit or add more operational types, therefore it is best practice to add all of the 7 operation types when setting up the template. Note that you can add only one type of an operation to the same template, i.e. all of the 7 operations, but not, for example, two "Install" operations.
    • Operation Type - select either of the following:
      • Install - installs an application or service on a VM.
      • Upgrade - upgrades apps/services on the VM according to the corresponding execution, etc. scripts.
      • Fix - deploys the fixes and patches.
      • Process Customizations - where applicable, deploys customizations of apps/services, e.g. .zip files.
      • Undeploy Customizations - deletes the previously installed customizations.
      • Startup - starts up the service; for more details, see the note in 4.3 Rejection of a Candidate.
      • Shutdown - shuts down the service; for more details, see the note in 4.3 Rejection of a Candidate.
  5. Click Operation Parameters, and PLACEHOLDER.
  6. Expand the folder, Orchestration > Execution > Pre-Execute, and enter the following:
    • Script Name - type in the script's name so that you could easily recognize it and identify its purpose, e.g. space_formatting.ps1; note that the file-name extension is .sh for Linux- and .ps1 for Windows-compatible VMs.
    • Required reboot - select the check-box to have the VM restarted after the completion of the script.
  7. Specify the execution and post-execution scripts as required, expand the Dependency Processing folder, and specify the scripts to be triggered if a VM is linked to any endpoint VM, i.e. if the template's virtual machine contains any dependent, endpoint-linked VMs, or cross-linked applications.
    Note: If you are using Linux, only bash-scripts are supported; if Windows, then only .ps scripts are supported. Supported PowerShell versions depend on the virtual machine being used.
  8. In the Service Template dialog box, expand the Service Management folder, click Deployment Profiles, Add Deployment Profile, and enter the following:
    Tip: When you are setting up a service template, especially a composite one, you need to set up a deployment group the template belongs to. A composite service template can include multiple deployment profiles which, in turn, can contain a number of deployment groups intended for a particular purpose. A deployment group is a collection of the same type of VMs to be provisioned. Note that the deployment group's settings take precedence over and supersede the settings of an individual service's OS.
    • Name - the profile's descriptive name.
    • Group Name - the deployment profile's name.
    • Key - type in the deployment profile's self-descriptive, meaningful definition or an ID (P3C automatically checks its uniqueness).
    • OS Platform and Hardware Platform - select the previously set-up platforms applicable to the profile.
    • Min Host Count - PLACEHOLDER.
    • Initial Host Count - the number of VMs to be deployed when the corresponding service is provisioned.
    • Max Host Count - PLACEHOLDER.
  9. Click OK.
